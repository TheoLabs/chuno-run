알림 기능 기획

slug: notification-feature 작성일: 2026-07-07 대상: B2C 웹 앱 사용자 상태: 초안

1. 배경 / 목적

문제. 현재 사용자는 자신과 관련된 중요 이벤트(내 글에 달린 댓글, 결제 완료 등)가 발생해도 이를 인지할 통로가 없어, 앱을 다시 열어야만 확인할 수 있다. 그 결과 반응이 늦고 재방문 동기가 약하다.

기대 효과. 중요 이벤트를 실시간으로 통지해 사용자의 즉각적 인지를 돕고, 이를 통해 재방문(리텐션)을 유도한다.

2. 목표

  • 사용자가 앱 내에서 자신과 관련된 알림을 한곳(알림센터)에서 모아 볼 수 있다.
  • 이벤트 발생 시 3초 이내에 인앱 알림이 표시된다.
  • 앱을 열지 않은 상태에서도 웹 푸시로 알림을 수신할 수 있다. (차기 마일스톤으로 연기, 2026-07-07)
  • 사용자가 알림을 읽음 처리하고, 수신 여부를 on/off로 제어할 수 있다.

3. 대상 사용자 / 이해관계자

구분대상관심사
주 사용자일반 소비자(B2C) 웹 앱 사용자내 이벤트를 놓치지 않고 빠르게 확인, 원치 않는 알림은 끄기
서비스 운영프로덕트 / 그로스 담당재방문율·알림 클릭률 향상
개발프론트엔드 / 백엔드실시간 전달 채널, 발송 인프라, 브라우저 푸시 권한 처리

4. 요구사항

Must인앱 알림센터

  • 헤더 등 공통 영역에 알림 진입점(벨 아이콘)과 미확인 개수 뱃지를 표시한다.
  • 알림 목록: 최신순 정렬, 항목별 유형(댓글/결제)·내용 요약·발생 시각·읽음 여부를 표시한다.
  • 이벤트 발생 시 3초 이내 알림센터/뱃지에 반영된다(실시간 채널 필요).
  • 알림 항목 클릭 시 관련 대상(해당 댓글 / 결제 상세)으로 이동한다.

Must알림 읽음 처리

  • 개별 알림 읽음 처리, 미확인 개수 감소 반영.
  • "모두 읽음" 일괄 처리.
  • 읽음 상태는 서버에 저장되어 기기/세션 간 일관되게 유지된다.

[변경 2026-07-07] 웹 푸시는 이번 범위에서 제외한다(차기 마일스톤으로 연기). 관련 요구사항 삭제.

Could알림 on/off 설정

  • 알림 유형별(댓글, 결제) 수신 여부를 켜고 끌 수 있다.
  • 인앱 채널 기준으로 유형별 수신 여부를 설정할 수 있다. (웹 푸시 채널 구분 설정은 웹 푸시 재개 시점의 차기 마일스톤 과제, 2026-07-07 변경)
  • off로 설정된 유형은 해당 채널로 전달하지 않는다.

5. 범위 (Scope)

포함

  • 인앱 알림센터 (목록·뱃지·이동)
  • 알림 읽음 처리 (개별 / 일괄)
  • 알림 유형별 on/off 설정
  • 이벤트 소스: 댓글, 결제 완료

제외 (Out of scope)

  • 이메일 알림
  • SMS 알림
  • 다국어(i18n) 대응
  • 모바일 네이티브 앱 푸시(iOS/Android 앱)
  • 웹 푸시 수신 (차기 마일스톤으로 연기, 2026-07-07 변경)
  • 댓글·결제 외 신규 이벤트 유형(추후 확장)

6. 성공 지표 / 완료 조건

항목기준
인앱 실시간성이벤트 발생 후 3초 이내 알림센터/뱃지에 표시
웹 푸시 수신 (차기 마일스톤)지원 브라우저에서 앱 미실행 상태에도 푸시 알림 수신 가능 — 이번 범위 제외, 차기 마일스톤으로 연기(2026-07-07)
읽음 처리개별/일괄 읽음이 서버에 반영되고 새로고침·타 세션에서도 유지
on/off 설정off한 유형이 해당 채널로 더 이상 전달되지 않음
이동 정확성알림/푸시 클릭 시 관련 대상 화면으로 정확히 진입

7. 리스크 / 고려사항

  • 실시간 채널. 3초 내 전달을 위해 WebSocket/SSE 등 상시 연결 또는 폴링 전략이 필요하다. 방식 선택이 인프라·비용에 영향.
  • (차기 마일스톤) 푸시 권한 거부. 웹 푸시가 재개되면 사용자가 브라우저 푸시 권한을 거부/무시할 수 있음 → 인앱 알림이 기본 안전망이 되어야 함. 이번 마일스톤은 웹 푸시 자체가 범위 제외(2026-07-07)이므로 해당 없음.
  • 브라우저 호환성. 일부 브라우저/사파리 버전은 Web Push 지원이 제한적 → 미지원 환경 UX(안내·graceful degradation) 정의 필요. (차기 마일스톤, 웹 푸시 재개 시 적용)
  • (차기 마일스톤) 중복 전달. 웹 푸시 재개 시 인앱과 웹 푸시가 동시에 울릴 때의 중복/우선순위 정책 필요(예: 앱 포그라운드 시 푸시 억제). 이번 마일스톤은 웹 푸시가 없어 해당 리스크 없음.
  • 알림 폭주. 짧은 시간 다수 이벤트 발생 시 묶음/집계(그룹핑) 정책 미정.
  • 보관·정리. 알림 보관 기간, 목록 페이지네이션/무한스크롤, 오래된 알림 정리 정책 필요.

8. 오픈 이슈

  • 실시간 전달 방식(WebSocket / SSE / 폴링) 중 무엇을 채택할지 — 아키텍처 결정 필요.
  • (차기 마일스톤) 웹 푸시 발송 인프라(자체 Web Push VAPID vs. FCM 등) 선정 — 2026-07-07 변경으로 이번 범위 제외, 웹 푸시 재개 시점에 재논의.
  • 알림 보관 기간 및 목록 최대 노출 개수 정책.
  • 동일/유사 알림 그룹핑(묶음) 여부와 규칙.
  • on/off 설정의 기본값(기본 on/off)과 채널별 세분화 수준.
  • (차기 마일스톤) 푸시 미지원 브라우저에서의 안내 문구/대체 UX — 웹 푸시와 함께 차기 마일스톤으로 연기(2026-07-07).
  • 알림 클릭 시 이동 경로(딥링크) 정의 — 댓글/결제 각각의 목적지 URL 규칙.
