에픽: 인앱 알림센터 feature:in-app-notification-center

기획서 4장 "인앱 알림센터"(Must). 벨 아이콘·뱃지·목록·이동을 3초 이내 실시간으로 반영한다.

NOTI-1

알림 데이터 모델 및 저장 API 설계

chore P0 S 인앱 알림센터
의존성:없음

설명: 유형(댓글/결제)·요약 내용·발생 시각·읽음 여부·대상 링크를 담는 알림 엔티티와 생성/조회 API를 만든다. 이후 모든 알림 기능(실시간 전달, 목록, 읽음 처리, on/off)이 이 모델 위에서 동작하므로 가장 먼저 확정한다.

  • 알림 레코드에 유형, 요약, 발생 시각, 읽음 여부, 대상 딥링크 필드가 저장된다.
  • 댓글/결제 이벤트 발생 시 알림 레코드가 생성되는 내부 API(or 이벤트 훅)가 있다.
  • 사용자별 알림 목록을 최신순으로 조회하는 API가 있다.
이동: 기획서 · 기능: 인앱 알림센터
NOTI-2

실시간 알림 전달 채널 구축

feature P0 M 인앱 알림센터
의존성:NOTI-1

설명: 이벤트 발생 후 3초 이내 클라이언트에 반영하기 위한 상시 연결(WebSocket/SSE) 또는 폴링 채널을 구축한다. 기획서 오픈 이슈: 방식(WebSocket vs SSE vs 폴링) 미확정 — 착수 전 아키텍처 결정 필요.

  • 이벤트 발생 후 3초 이내 클라이언트가 새 알림을 수신한다(측정 가능).
  • 연결 끊김/재접속 시에도 유실 없이 알림이 전달되거나 목록 재조회로 보완된다.
  • 전달 방식 선택 근거와 인프라 비용이 문서화된다.
이동: 기획서(리스크 7장, 오픈이슈 8장)
NOTI-3

헤더 벨 아이콘 + 미확인 개수 뱃지

feature P0 S 인앱 알림센터
의존성:NOTI-2

설명: 공통 헤더 영역에 알림 진입점(벨 아이콘)과 미확인 개수 뱃지를 표시한다. 실시간 채널(NOTI-2)로부터 이벤트를 받아 즉시 뱃지 숫자를 갱신한다.

  • 헤더에 벨 아이콘이 표시되고 클릭 시 알림 목록(NOTI-4)이 열린다.
  • 미확인 알림이 있으면 뱃지에 정확한 개수가 표시된다.
  • 새 이벤트 발생 시 새로고침 없이 뱃지 숫자가 갱신된다.

NOTI-4

알림 목록 UI (알림센터)

feature P0 M 인앱 알림센터
의존성:NOTI-1

설명: 알림 목록을 최신순으로 정렬해 보여준다. 항목별로 유형(댓글/결제), 내용 요약, 발생 시각, 읽음 여부를 표시한다. 목록 최대 노출 개수/페이지네이션 정책은 기획서 오픈 이슈로 미정이므로 우선 기본 목록(예: 최근 N건)으로 구현하고 정책 확정 시 후속 조정한다.

  • 알림이 발생 시각 최신순으로 정렬되어 표시된다.
  • 각 항목에 유형·요약·시각·읽음 여부가 노출된다.
  • 미확인 항목이 시각적으로 구분된다.
이동: 기획서 4장
NOTI-5

알림 클릭 시 대상 화면으로 이동 (딥링크)

feature P1 S 인앱 알림센터
의존성:NOTI-4

설명: 알림 항목 클릭 시 해당 댓글 또는 결제 상세로 이동한다. 기획서 오픈 이슈: 댓글/결제 각각의 목적지 URL(딥링크) 규칙 미정 — 규칙 확정이 선행되어야 한다. (이 규칙은 차기 마일스톤에서 웹 푸시가 재개될 경우 클릭 진입 흐름에도 재사용될 예정이다.)

  • 댓글 알림 클릭 시 해당 댓글 위치로 정확히 이동한다.
  • 결제 알림 클릭 시 해당 결제 상세로 정확히 이동한다.
  • 이동과 동시에 해당 알림이 읽음 처리된다(NOTI-6 연계).
이동: 기획서 8장 오픈 이슈

에픽: 알림 읽음 처리 feature:notification-read-status

기획서 4장 "알림 읽음 처리"(Must). 개별/일괄 읽음과 서버 저장을 통한 세션 간 일관성을 보장한다.

NOTI-6

개별 알림 읽음 처리

feature P0 S 알림 읽음 처리
의존성:NOTI-4

설명: 알림 항목을 개별적으로 읽음 처리하고, 처리 즉시 미확인 개수(NOTI-3 뱃지)에 반영한다. 읽음 상태는 서버에 저장한다.

  • 알림 항목을 읽음 처리하면 서버에 상태가 저장된다.
  • 읽음 처리 즉시 헤더 뱃지의 미확인 개수가 감소한다.
  • 이미 읽은 알림은 목록에서 읽음 상태로 구분 표시된다.
이동: 기획서 4장
NOTI-7

모두 읽음 일괄 처리

feature P1 S 알림 읽음 처리
의존성:NOTI-6

설명: "모두 읽음" 버튼으로 사용자의 모든 미확인 알림을 한 번에 읽음 처리한다.

  • "모두 읽음" 클릭 시 목록의 모든 미확인 알림이 읽음 상태로 바뀐다.
  • 헤더 뱃지의 미확인 개수가 0으로 바뀐다.
  • 서버 반영 실패 시 사용자에게 오류가 안내된다.
이동: 기획서 4장
NOTI-8

읽음 상태 세션/기기 간 일관성 검증

chore P1 S 알림 읽음 처리
의존성:NOTI-6

설명: 성공 지표("읽음 처리가 서버에 반영되고 새로고침·타 세션에서도 유지")를 보장하기 위한 검증 및 필요 시 캐시/동기화 로직 보완. 다른 브라우저·기기로 로그인해도 동일한 읽음 상태가 보이는지 확인한다.

  • 한 기기에서 읽음 처리한 알림이 다른 기기/세션에서도 읽음으로 보인다.
  • 새로고침 후에도 읽음 상태가 유지된다.
  • 동시성(같은 알림을 두 세션에서 동시 읽음 처리) 시 오류 없이 일관된 상태로 수렴한다.

에픽: 알림 on/off 설정 feature:notification-preferences

기획서 4장 "알림 on/off 설정"(Could). 이번 마일스톤은 인앱 채널 기준으로 유형별 수신 제어를 제공한다. 웹 푸시 채널 설정은 웹 푸시 자체가 차기 마일스톤으로 연기(2026-07-07)됨에 따라 함께 연기됐다.

NOTI-13

알림 유형별 on/off 설정 UI 및 API (인앱 채널)

feature P2 M 알림 on/off 설정
의존성:NOTI-1

설명: 사용자가 댓글/결제 유형별로 인앱 수신 여부를 설정할 수 있는 화면과 저장 API를 만든다. 기획서 오픈 이슈: 기본값(기본 on/off) 미정 — 우선 모든 유형 기본 on으로 구현한다. 웹 푸시 채널 설정은 차기 마일스톤 연기(2026-07-07)로 이번 범위에서 제외하며, 설정 화면에는 "차기 제공 예정"으로 비활성 노출한다.

  • 댓글/결제 각각의 인앱 수신 여부를 개별적으로 켜고 끌 수 있다.
  • 설정이 서버에 저장되어 새로고침/재로그인 후에도 유지된다.
  • 웹 푸시 채널 옵션은 선택 불가(disabled)로 노출되며 "차기 제공 예정" 라벨이 표시된다.
  • 미확정 정책(기본값)은 문서화된 가정으로 표시된다.
이동: 기획서 4장, 8장 오픈 이슈
NOTI-14

off 설정 유형에 대한 발송 차단 적용 (인앱 채널)

feature P2 S 알림 on/off 설정
의존성:NOTI-13 · NOTI-2

설명: NOTI-13에서 저장된 설정을 실시간 전달(NOTI-2) 경로에 적용해, off된 유형의 인앱 알림이 전달되지 않도록 한다. 성공 지표("off한 유형이 해당 채널로 더 이상 전달되지 않음")를 인앱 채널 기준으로 충족한다. 웹 푸시 발송 차단은 차기 마일스톤에서 웹 푸시가 재개될 때 함께 구현한다.

  • 인앱 알림을 off한 유형은 알림센터/뱃지에 나타나지 않는다.
  • 설정 변경 후 즉시(다음 이벤트부터) 반영된다.
이동: 기획서 6장 성공 지표

NOTI-9

웹 푸시 발송 인프라 선정 및 서버 기반 구축

chore P1 L 웹 푸시 차기 마일스톤 연기(2026-07-07)
의존성:없음

설명: 기획서 오픈 이슈: 자체 Web Push(VAPID) vs FCM 등 인프라 미선정 — 우선 조사·의사결정 후 발송 서버(구독 저장, 페이로드 발송, 재시도)를 구축한다. 규모가 커 별도 스파이크/설계 검토를 권장한다.

  • Web Push 발송 방식(VAPID 자체 구현 vs FCM 등)이 결정되고 근거가 문서화된다.
  • 서버에서 구독 정보를 저장하고 대상 사용자에게 푸시 페이로드를 발송할 수 있다.
  • 발송 실패(만료된 구독 등)에 대한 기본 처리(삭제/재시도)가 있다.
이동: 기획서 8장 오픈 이슈
NOTI-10

브라우저 푸시 권한 요청 및 구독 등록/해제

feature P1 M 웹 푸시 차기 마일스톤 연기(2026-07-07)
의존성:NOTI-9

설명: 클라이언트에서 브라우저 푸시 권한을 요청하고, 허용 시 구독을 서버(NOTI-9)에 등록한다. 사용자가 원할 경우 구독을 해제할 수 있다.

  • 사용자가 푸시 권한을 허용하면 구독 정보가 서버에 등록된다.
  • 사용자가 구독을 해제하면 서버에서도 해당 구독이 제거된다.
  • 권한 거부 시 인앱 알림(NOTI-3, NOTI-4)은 정상 동작한다(안전망).

NOTI-11

Service Worker 푸시 수신 및 클릭 시 화면 진입

feature P1 M 웹 푸시 차기 마일스톤 연기(2026-07-07)
의존성:NOTI-10 · NOTI-5

설명: Service Worker에서 푸시 메시지를 수신해 브라우저 알림으로 표시하고, 앱 미실행 상태에서도 동작하도록 한다. 클릭 시 NOTI-5의 딥링크 규칙을 재사용해 관련 화면으로 진입한다. 인앱 포그라운드 상태에서의 중복 알림 억제 정책(NOTI-2)도 함께 적용한다.

  • 앱을 열지 않은 상태에서 이벤트 발생 시 브라우저 푸시 알림을 수신한다.
  • 푸시 알림 클릭 시 해당 댓글/결제 상세 화면으로 진입한다.
  • 앱이 포그라운드일 때는 정책에 따라 중복 알림이 억제된다.
이동: 기획서 7장 리스크(중복 전달)
NOTI-12

푸시 미지원 브라우저 안내 UX

feature P2 S 웹 푸시 차기 마일스톤 연기(2026-07-07)
의존성:NOTI-10

설명: 표준 Web Push(Service Worker)를 지원하지 않는 브라우저/OS에서 사용자에게 안내 문구를 보여주고 인앱 알림으로 대체되도록 graceful degradation을 적용한다. 기획서 오픈 이슈: 정확한 안내 문구/대체 UX 미정이므로 기본 안내 문구로 우선 구현한다.

  • 미지원 브라우저에서는 푸시 권한 요청 UI 대신 안내 메시지가 표시된다.
  • 미지원 환경에서도 인앱 알림센터는 정상 동작한다.
이동: 기획서 7장 리스크(브라우저 호환성)

이번 범위 제외
